================================================================================
                            API ENDPOINTS (7 total)
================================================================================

1. POST   /coaches/{coach_id}/availability
   Set weekly recurring availability with time slots

2. POST   /coaches/{coach_id}/exceptions
   Add one-time exception or override

3. GET    /coaches/{coach_id}/slots
   Get available slots for a specific date

4. POST   /bookings
   Create a new booking (safe, idempotent)

5. GET    /users/{user_id}/bookings
   Get all bookings for a user

6. PUT    /bookings/{id}
   Modify an existing booking

7. DELETE /bookings/{id}
   Cancel a booking

================================================================================
                          DATABASE SCHEMA (5 tables)
================================================================================

users
├── id, name, timezone, created_at, updated_at, deleted

coaches
├── id, name, timezone, created_at, updated_at, deleted

availability
├── id, coach_id, day_of_week, start_time, end_time
├── FOREIGN KEY (coach_id) → coaches.id
├── UNIQUE (coach_id, day_of_week) per slot
└── Indexes: coach_id, deleted

availability_exceptions
├── id, coach_id, date, start_time, end_time, is_available
├── FOREIGN KEY (coach_id) → coaches.id
├── UNIQUE (coach_id, date)
└── Indexes: coach_id, date, deleted

bookings
├── id, user_id, coach_id, start_time, end_time, status, idempotency_key
├── FOREIGN KEY (user_id) → users.id
├── FOREIGN KEY (coach_id) → coaches.id
├── UNIQUE (coach_id, start_time) ← Prevents double booking
├── UNIQUE (idempotency_key)
└── Indexes: user_id, coach_id, start_time, status, deleted

================================================================================
                        CONCURRENCY SAFETY APPROACH
================================================================================

Multi-Layer Defense:
1. Application Layer: Check availability before booking
2. Transaction Layer: Use BEGIN/COMMIT for atomicity
3. Row Lock Layer: FOR UPDATE to prevent race conditions
4. Constraint Layer: UNIQUE(coach_id, start_time) for guarantee
5. Idempotency: Prevent duplicate bookings with key

Result: ZERO chance of double booking ✅

                        HOW TO RUN THE PROJECT
================================================================================

1. START DATABASE:
   docker-compose up -d

2. RUN MIGRATIONS:

3. BUILD & RUN:
   make run
   # OR
   go build -o bin/booking-api-service cmd/main.go
   ./bin/booking-api-service

   http://localhost:8080
